# MVP 1 구현 기록

작성일: 2026-07-17  
대상: 당신의 독자 Pt.3 — 기본 인프라 및 핵심 기능 개발

## 목표

사용자가 계정을 만들고 로그인한 뒤, 방해 요소가 적은 화면에서 글을 작성하고 안전하게 저장할 수 있는 최소 기능 제품을 만드는 것이 목표였다.

구현 범위는 다음 네 가지로 제한했다.

1. 기존 Next.js 프로젝트를 실행 가능한 상태로 정리한다.
2. Supabase Auth로 이메일 회원가입과 로그인을 제공한다.
3. 사용자별 글 생성·조회·수정·삭제를 연결한다.
4. 미니멀 에디터와 자동 저장을 구현한다.

AI 감정 분석과 독자 페르소나 응답 생성은 이번 단계의 핵심인 글쓰기·저장 기반이 안정된 이후 진행하도록 범위에서 제외했다.

## 시작 상태와 판단

저장소에는 Next.js App Router 프로젝트와 Supabase 초기 스키마가 이미 있었다. 따라서 프로젝트를 새로 생성하지 않고 기존 구조를 유지하면서 필요한 기능을 추가했다.

기존 화면과 모델 파일의 한글 문자열 일부가 깨져 있었고 JSX 문법도 손상되어 있었다. 부분 수정으로 오류를 누적시키는 대신, 현재 단계에 필요한 인증·서재·에디터 흐름을 중심으로 화면을 다시 구성했다.

## 구현 방식

### 1. Supabase 연결

브라우저와 서버 환경을 구분해 Supabase 클라이언트를 구성했다.

- 브라우저 클라이언트: 로그인, 회원가입, 글 CRUD 처리
- 서버 클라이언트: 향후 Server Component와 Route Handler에서 사용할 수 있는 쿠키 기반 연결
- 미들웨어: 만료된 인증 토큰을 갱신하고 갱신된 쿠키를 요청과 응답에 반영

프론트엔드에는 공개 가능한 Supabase 프로젝트 URL과 Publishable Key만 전달한다. `service_role` 키는 사용하지 않는다.

필요한 환경 변수는 다음과 같다.

```env
NEXT_PUBLIC_SUPABASE_URL=https://your-project.supabase.co
NEXT_PUBLIC_SUPABASE_PUBLISHABLE_KEY=your-publishable-key
```

환경 변수가 없을 때 애플리케이션이 빌드 중 실패하지 않도록 클라이언트를 지연 생성하고, 화면에는 연결 방법을 안내한다.

### 2. 인증

로그인되지 않은 사용자는 이메일과 비밀번호를 이용하는 인증 화면을 보게 된다.

- 기존 사용자는 `signInWithPassword`로 로그인한다.
- 신규 사용자는 `signUp`으로 가입한다.
- 이메일 확인이 활성화된 프로젝트에서는 확인 링크 안내를 표시한다.
- 로그인 상태 변화는 Auth 이벤트를 구독해 즉시 화면에 반영한다.

서버 측 세션 갱신에서는 저장된 세션 객체를 그대로 신뢰하지 않고 Supabase가 권장하는 `getClaims()`를 호출한다.

### 3. 데이터베이스 보안

글은 `entries` 테이블에 저장하며 모든 요청에 `user_id`를 포함한다. 데이터베이스에서는 RLS 정책이 현재 로그인한 사용자의 `auth.uid()`와 글의 `user_id`가 같은 경우에만 접근을 허용한다.

신규 Auth 사용자가 생성되면 `profiles` 행을 자동으로 생성하는 트리거를 추가했다. 트리거 함수는 외부에 노출되지 않는 `private` 스키마에 두고 실행 권한을 회수했다.

`authenticated` 역할에는 필요한 최소 범위만 부여했다.

- 활성 페르소나 조회
- 자신의 글 조회, 생성, 수정, 삭제

RLS가 최종적으로 행 단위 소유권을 검사하므로 다른 사용자의 글 ID를 알아도 접근할 수 없다.

### 4. 글 CRUD

로그인 후 왼쪽 서재에서 사용자의 글을 최근 수정 순으로 불러온다.

- 새 글: 빈 에디터를 열고 첫 입력 시 레코드를 생성한다.
- 조회: 서재 항목을 누르면 제목과 본문을 에디터로 불러온다.
- 수정: 제목 또는 본문이 바뀌면 기존 레코드를 갱신한다.
- 삭제: 확인 대화상자를 거친 뒤 레코드를 삭제한다.

서재 목록은 저장이 완료될 때마다 다시 불러와 최신 수정 시각과 제목을 보여준다.

### 5. 자동 저장

키 입력마다 네트워크 요청을 보내지 않도록 800ms 디바운스를 적용했다.

1. 제목이나 본문이 변경되면 저장 상태를 `저장 중`으로 바꾼다.
2. 마지막 입력 이후 800ms 동안 추가 입력이 없으면 저장 요청을 보낸다.
3. 아직 글 ID가 없으면 `insert`, 있으면 `update`를 실행한다.
4. 성공하면 마지막 저장 값을 기록하고 `저장됨`을 표시한다.
5. 실패하면 화면에 저장 실패 상태를 표시한다.

마지막으로 저장된 제목과 본문을 별도로 기억해 값이 실제로 달라진 경우에만 요청을 보낸다.

### 6. 몰입형 에디터

화면은 서재 사이드바와 종이 형태의 에디터로 단순화했다.

- 에디터 내부의 버튼과 도구 모음을 최소화했다.
- 제목과 본문 입력에 읽기 편한 세리프 계열 글꼴을 사용했다.
- 글자 수와 저장 상태만 상단에 작게 표시했다.
- 모바일에서는 사이드바를 위쪽 목록으로 전환하고 종이 여백을 줄였다.

## 검증

`npm run build`를 실행해 다음 항목을 확인했다.

- Next.js 프로덕션 컴파일 성공
- TypeScript 타입 검사 성공
- 정적 페이지 생성 성공
- 미들웨어 번들 생성 성공

`git diff --check`로 잘못된 공백이나 패치 오류가 없는 것도 확인했다.

`npm audit --omit=dev`에서는 Next.js가 참조하는 PostCSS 경로에 중간 등급 경고 2건이 보고되었다. 제시된 자동 수정은 Next.js 9로 강제 다운그레이드하는 파괴적 변경이므로 적용하지 않았다. 프레임워크의 호환 가능한 패치가 제공될 때 다시 검토한다.

## 변경된 주요 파일

- `app/page.tsx`: 인증, 서재, CRUD, 자동 저장 에디터

- `lib/supabase/proxy.ts`: 인증 토큰 갱신
- `middleware.ts`: 요청별 세션 갱신 연결
- `supabase/migrations/202607160001_initial_schema.sql`: RLS, 권한, 프로필 생성 트리거
- `.env.example`: 필요한 환경 변수 예시

## 배포 전 남은 작업

1. Supabase 프로젝트를 만들고 `.env.local`에 프로젝트 URL과 Publishable Key를 입력한다.
2. 초기 마이그레이션을 Supabase 프로젝트에 적용한다.
3. 이메일 인증의 Site URL과 Redirect URL을 운영 도메인에 맞춘다.
4. 같은 환경 변수를 Vercel 프로젝트에 등록한다.
5. Vercel 배포 후 회원가입, 로그인, 자동 저장, 다른 계정 간 데이터 격리를 직접 확인한다.
